## Project Summary

This project builds an end-to-end MLOps pipeline for detecting and responding to data drift in a credit card fraud detection system. Starting from the [Kaggle Credit Card Fraud Detection dataset](https://www.kaggle.com/datasets/mlg-ulb/creditcardfraud) (284,807 transactions, 0.173% fraud rate), we train a baseline XGBoost classifier, evaluate it against five simulated production batches with varying drift profiles, and provide a Streamlit monitoring dashboard with automated retraining recommendations.

From the **project root** directory:

```bash

```

### Dashboard Features

- **Verdict Card** — automated health status (stable / warning / critical) with actionable recommendations, powered by the tiered decision logic in `src/retrain_trigger.py`
- **Snapshot Metrics** — AUC-ROC, PR-AUC, Precision, Recall, F1 with deltas from baseline
- **Drift Evidence** — top 15 features ranked by PSI with warning/critical threshold lines, plus Evidently AI cross-validation
- **Performance Impact** — AUC-ROC and PR-AUC trend across all batches with degradation shading
- **Confusion Matrix Comparison** — baseline vs. selected batch side-by-side
- **Feature Inspector** — interactive feature selector with KDE distribution overlay (training vs. production) and per-feature PSI/KS statistics

Use the sidebar dropdown to switch between the five production batches.

## Methodology

### Pipeline Overview

1. **EDA** — Identified severe class imbalance (1:577), top discriminative features (V4, V11, V14, V17, V12, V10) via KS statistics and correlation analysis, and temporal/amount patterns.

2. **Baseline Model** — XGBoost classifier with SMOTE oversampling (`sampling_strategy=0.5`), trained on an 80/20 split. Metrics logged: F1, AUC-ROC, Precision, Recall, PR-AUC.

3. **Drift Detection** — Three statistical tests applied per feature across all five production batches:
   - **KS Test** — measures maximum distributional divergence (p < 0.05 = flagged)
   - **PSI** — Population Stability Index (< 0.10 stable, 0.10–0.20 moderate, > 0.20 critical)
   - **KL Divergence** — information-theoretic distance measure
   - **Evidently AI** — independent validation using Jensen-Shannon divergence

4. **Impact Analysis** — Frozen model evaluated on each drift batch. Performance degradation quantified as deltas from baseline and traced back to drift scores.

5. **Monitoring & Retraining Strategy** — Automated drift check implemented in `src/drift_monitor.py` (runnable via CLI or import). Retraining decisions follow a tiered logic in `src/retrain_trigger.py`:
   - **Tier 1 (Critical):** AUC-ROC < 0.5 — model inverting predictions
   - **Tier 2 (Critical):** Any feature fails both KS and PSI critical thresholds
   - **Tier 3 (Warning):** AUC-ROC drop > 0.10 without individual feature spikes
   - **Tier 4 (Warning):** ≥ 3 features at PSI warning or > 20 KS-flagged
   - **Stable:** No triggers activated

## Key Findings

| Batch | Drift Type | AUC-ROC | Verdict |
|-------|-----------|---------|---------|
| **drift_1** | Localised covariate drift (V1 spike, PSI=2.05) | ~0.51 | 🚨 Retrain — single feature collapse breaks the model |
| **drift_2** | Feature distribution shift (Amount, PSI=2.73) | ~0.48 | 🚨 Retrain — model inverts predictions |
| **drift_3** | Label drift (3× fraud rate) | ~0.92 | ⚠️ Recalibrate threshold, do not retrain |
| **drift_4** | Silent distributed drift (no PSI/KS actionable flags) | 0.4480 | 🚨 Retrain — model ranking inverted; missed all ~25 frauds; Layer 1 blind spot |
| **drift_5** | Label drift only (11.5× fraud rate, patterns stable) | ~0.91 | ⚠️ Warning (KS flags 28 features); recalibrate threshold, do not retrain |

### Critical Insight

**Standard single-feature PSI monitoring would have caught drift_1 and drift_2 but completely missed drift_4** — the most dangerous batch. drift_4 demonstrates that small, coordinated shifts across many features can be more destructive than a large spike in one feature. This finding underscores the need to combine distribution-level tests with model performance monitoring (AUC-ROC tracking) for robust production systems.

## Running the Drift Monitor Standalone

The drift detection module can be used independently of the dashboard:

```bash
# Run drift detection on a batch
python src/drift_monitor.py --batch data/drift_1.csv --artifacts notebooks/

# Run the full retraining trigger evaluation
python src/retrain_trigger.py --batch data/drift_1.csv --key drift_1 --artifacts notebooks/
```
